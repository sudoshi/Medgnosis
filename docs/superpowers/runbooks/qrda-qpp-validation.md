# QRDA And QPP Validation Runbook

Last updated: 2026-06-26

## Purpose

This runbook defines the current validation path for Medgnosis QRDA Category I,
QRDA Category III, and QPP JSON reporting artifacts. It separates local fixture
and structural validation from official external validation.

## Current Validator Targets

- QRDA Cat I/Cat III: Cypress v7.5 with integrated CVU+, released 2026-06-18,
  supporting 2026 and 2027 reporting-year eCQMs.
- QPP JSON: CMS QPP Submissions API developer documentation and sandbox/API
  access path.

Primary references:

- https://cypress.healthit.gov/cypress/release.html
- https://developer.cms.gov/qpp-api/
- https://cmsgov.github.io/qpp-submissions-docs/

## Local Fixtures

Generated by:

```bash
npx tsx apps/api/scripts/quality-reporting-samples.ts
```

Current outputs:

- `apps/api/test-fixtures/quality/qrda-cat1-sample.xml`
- `apps/api/test-fixtures/quality/qrda-cat3-sample.xml`
- `apps/api/test-fixtures/quality/qpp-submission-sample.json`

The fixtures are deterministic and intentionally small. They prove that the
current serializers emit well-formed local artifacts. They do not prove CMS
submission readiness by themselves.

## Local Validation Commands

QRDA structural smoke:

```bash
./scripts/qrda-validate.sh
```

QPP structural smoke:

```bash
./scripts/qpp-validate.sh
```

Expected local behavior:

- The sample generator rewrites the three fixtures.
- QRDA XML parses as CDA `ClinicalDocument` and contains the expected measure
  section/template markers.
- QPP JSON contains `performanceYear`, a quality `measurementSets` entry,
  `submissionMethod: electronicHealthRecord`, and non-negative integer measure
  counts.
- If no external command is configured, the scripts print an explicit skip for
  official CVU+/QPP validation and still pass the local structural gate.

## Official QRDA CVU+ Handoff

When Cypress CVU+ is installed in a CI runner or operator workstation, wire it
through these script hooks:

```bash
QRDA_CVU_CAT1_CMD='your-cvu-cat1-command "$QRDA_FILE"' \
QRDA_CVU_CAT3_CMD='your-cvu-cat3-command "$QRDA_FILE"' \
./scripts/qrda-validate.sh
```

The scripts export `QRDA_FILE` separately for Cat I and Cat III. The command
must select the reporting program, document type, and reporting year expected by
the installed Cypress/CVU+ package.

Do not mark QRDA submission-ready until:

- Cat I passes CVU+ for the target reporting year.
- Cat III passes CVU+ for the target reporting year.
- Measure-specific QDM patient-data entries are complete enough for the selected
  measure/test deck.
- The exact Cypress/CVU+ version and reporting-year IG are recorded in the
  release evidence.

## Official QPP Handoff

When QPP sandbox/API access is available, wire it through:

```bash
QPP_VALIDATE_CMD='your-qpp-sandbox-command "$QPP_FILE"' ./scripts/qpp-validate.sh
```

The script exports `QPP_FILE` with the deterministic sample JSON. The external
command must handle authentication and sandbox endpoint selection.

Do not mark QPP submission-ready until:

- The JSON passes the official schema/sandbox for the target performance year.
- The selected `measureId` values match the current QPP measure inventory.
- Real program, TIN/NPI, eligibility, and submitter authorization requirements
  are satisfied outside the local fixture.
